# AnyCapsule

A Chrome extension that adds context management, message tools, and cross-platform handoff to Claude, ChatGPT, and Google Gemini.

---

## Features

| Feature | Description |
|---|---|
| **Context Extraction** | Scans the active conversation and surfaces key decisions, next steps, code blocks, and topic keywords in a collapsible side panel |
| **Pinboard** | Pin any message from any role; pins persist across page reloads and are scoped to each conversation |
| **Soft Delete** | Hide messages from view without removing them from the AI's context; toggle visibility at any time; bulk-delete mode for selecting multiple messages at once |
| **Local Edit** | Inline-edit any message in the DOM with a textarea widget; saves an edit history of up to 10 versions; one-click revert to original |
| **Text Highlights** | Select any text in a message and highlight it in yellow, green, or red; highlights survive page reload via XPath re-anchoring |
| **Context Handoff** | Packages conversation context into a structured prompt (decisions + code + recent exchange) that can be copied to clipboard, pinned, or sent to a new tab on a different platform |
| **Token Limit Banner** | Detects when a platform's context limit is approaching and surfaces a one-click handoff banner |

## Installation (Development)

**Prerequisites:** Node.js 18+, Chrome or Chromium

```bash
# Install dependencies
npm install

# Build in watch mode (rebuilds on every file change)
npm run dev
```

Then load the extension in Chrome:

1. Open `chrome://extensions`
2. Enable **Developer mode** (top-right toggle)
3. Click **Load unpacked**
4. Select the `dist/` folder

After any source change, the watcher rebuilds automatically. Click the reload icon next to the extension in `chrome://extensions` to pick up the new build (or press `Ctrl+R` if using a hot-reload plugin).

## Architecture

### Adapter Pattern

Every platform exposes a different DOM structure. The adapter layer normalises this behind a common interface so all feature code is platform-agnostic:

```
PlatformAdapter (baseAdapter.js)
  ├── getChatContainer()       → Element | null
  ├── getMessageElements()     → Element[]
  ├── extractMessageData(el)   → { messageId, role, text, element }
  ├── getConversationId()      → string
  ├── getPlatformIdentifier()  → 'claude' | 'chatgpt' | 'gemini'
  └── detectTokenLimitWarning()→ boolean
```

Selector strategy (all three adapters follow the same priority order):
1. Semantic / ARIA attributes (`role="log"`, `data-message-author-role`)
2. `data-testid` attributes
3. Known class fragments (last resort; least stable)

### Event Bus

Features communicate through DOM custom events rather than direct imports, which avoids circular dependencies and keeps features loosely coupled:

| Event | Payload | Fired when |
|---|---|---|
| `lms:adapterReady` | `{ adapter, platform, conversationId }` | Chat container found; adapter initialised |
| `lms:messageAdded` | `{ messageId, role, text, element }` | New message detected by MutationObserver |
| `lms:tokenLimitWarning` | `{ platform, conversationId }` | Token limit pattern detected |

### Storage Schema

All data is stored in `chrome.storage.local` under namespaced keys:

```
lms::<platform>::<conversationId>::<type>
```

Where `<type>` is one of: `pin`, `highlight`, `edit`, `deleted`, `handoff`, `meta`.

A self-imposed 2 MB budget is enforced with:
- Warning log at 80% usage
- Automatic LRU eviction at 90% usage (oldest entries removed first)

### SPA Navigation

Claude and ChatGPT are SPAs that navigate without full page reloads. The content script detects URL changes by intercepting `history.pushState` and listening to `popstate`, then re-initialises the adapter and all features for the new conversation.

### MutationObserver

A debounced `MutationObserver` watches the chat container for new messages. The 400ms debounce collapses rapid streaming updates (dozens of mutations per second while the AI is typing) into a single processing pass.

## Context Extraction

The `contextExtractor` service runs five analysis passes over the messages returned by an adapter:

1. **Code block extraction** — fenced (` ``` `) and inline (`` ` ``) via regex
2. **Decision detection** — regex patterns matching conclusions and agreements (e.g. "let's go with", "we've decided", "the solution is")
3. **Next-step detection** — patterns matching action items (e.g. "next:", "step N", "you should now")
4. **Topic mining** — capitalised noun phrases + a hardcoded domain keyword list (api, database, docker, react, etc.), sorted by frequency
5. **Message condensation** — the last 6 messages kept verbatim; older messages truncated to 300 characters

The output is an `ExtractedContext` object that feeds both the Context Side Panel UI and the Handoff prompt builder.

### Handoff Prompt Structure

The generated handoff prompt follows a structured template:

```
[CONTEXT SUMMARY]   — platform, message counts, topics
[KEY DECISIONS]     — up to 10 detected decision sentences
[CODE]              — up to 5 fenced code blocks (truncated at 1500 chars each)
[NEXT STEPS]        — up to 8 detected action sentences
[RECENT EXCHANGE]   — last 6 messages verbatim
[CONFIRMATION REQUEST] — asks the receiving AI to summarise its understanding
```

## Permissions

| Permission | Why |
|---|---|
| `activeTab` | Read the URL and inject scripts into the active tab |
| `storage` | Persist pins, edits, highlights, and deleted messages across sessions |
| `scripting` | Programmatically inject content scripts when needed |
| `clipboardWrite` | Copy handoff prompts and code blocks to clipboard |
| `tabs` | Query the active tab's URL for platform detection |

Host permissions are scoped to exactly the four supported URLs — no broad `<all_urls>` access.

## Known Limitations

- **Soft delete is visual only.** Hiding a message removes it from the user's view but the AI's context window is unchanged — the platform still sends it to the model.
- **Local edits are visual only.** Editing a message in the DOM does not change what the AI sees. The platform's copy of the conversation is unaffected.
- **Handoff injection into React inputs** requires a workaround for React's synthetic event system. Setting `textarea.value` directly does not trigger React state updates in Claude or ChatGPT. The clipboard delivery method is the reliable fallback.
- **Selector fragility.** The extension depends on DOM selectors that can break when Claude, ChatGPT, or Gemini push frontend updates. The adapter fallback chains are designed to degrade gracefully, but a significant redesign on any platform may require selector updates.
- **Message IDs.** When platform-provided IDs (`data-testid`, `data-message-id`) are absent, IDs fall back to `<platform>::<convId>::<index>`. Index-based IDs are not stable if the platform reorders or injects messages.
- **Gemini Shadow DOM.** Gemini uses Angular custom elements with Shadow DOM. The adapter pierces one level of shadow root; deeply nested shadow trees may not be reached.
- **`document.execCommand` deprecation.** The fallback text-injection path uses the deprecated `execCommand('insertText')` API. This may stop working in future Chrome versions.

## Development Notes

### Adding a New Platform

1. Create `src/adapters/<name>Adapter.js` extending `PlatformAdapter`
2. Implement all six interface methods (see `baseAdapter.js`)
3. Add hostname detection in `src/content.js` and `src/popup.js`
4. Add the new URL to `manifest.json` under `host_permissions` and `content_scripts.matches`

### Storage Quota

The default quota is 2 MB (self-imposed; Chrome's hard limit is 10 MB). To raise it, change `QUOTA_BYTES` in `src/services/storage.js`. The LRU eviction threshold scales with the quota automatically.

### Debugging

All log messages are prefixed with `[LM-Source]` or `[LM-Source][<Module>]`. Open DevTools on any supported page and filter by `LM-Source` to see the full initialisation sequence, message processing, and storage operations.
